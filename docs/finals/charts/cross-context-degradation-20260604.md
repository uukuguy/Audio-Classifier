## 关键发现 (D-26 应对论据)

### 退化曲线相对平缓 (vs 我们事前估计)

事前估计 R4 全栈复赛跌 **0.05-0.15**, 实测 ctx 单源只在 < 5s 才大跌, 10s 上下文只跌 **0.029** = 我们的滚动窗特征工程 (10/25/50/100/200) 在短上下文下仍有信号.

### 五类敏感度分级

| 类 | 敏感度 | 解释 |
|---|---|---|
| C | 高 (-0.114 @ 1s) | C 是连续说话, 强烈依赖最近上下文; 短上下文下"是否在说话" 信号弱 |
| T | 高 (-0.242 @ 1s) | T = turn change, 需要看到"前一个 turn 已经结束" 才能预测下一 turn |
| BC | N/A (评估集 0 BC F1) | BC 太稀疏, N=7 不能下结论. 推测中等敏感 |
| I | 低 (~ -0.07 @ 1s) | I = interruption, 跟历史模式弱相关 |
| NA | 极低 (-0.001 @ 1s) | NA = silence, 主要看末段连续性, 短上下文够用 |

### R4 全栈推算 (基于 ctx 在 R4 中权重)

R4 = NSOTA_07 + e2v_ms 0.03 + hub_ms 0.03 = orthofuse(ctx/wsp/hub) + 软加 wsp_ms + e2v_ms + hub_ms.

ctx 在 R4 中只承担:
- C 类 100% (orthofuse 路由)
- BC 类 100%
- NA 类 100%
- T 类: orthofuse 给 ctx 0% (T = 0.7 wsp + 0.3 hub), 之后软加才动 T 列
- I 类: orthofuse 给 ctx 1/3 (ctx+wsp+hub 三投票)

ctx 短上下文退化主要影响 C 和 NA 列 (T/I 由 SSL 头主导). C 在 R4 中稳定 (~0.97), NA 不敏感.

→ **R4 估短上下文退化 = ctx 退化 × 平均权重 ≈ ctx 退化 × 0.5**

## 答辩用法

### 当评委问"复赛动态时长怎么处理"
"我们做了内部 cross-context probe (slide N), 实测 ctx 单源在 10s 上下文仅退化 3%, 5s 退化 5%. R4 全栈因为 SSL_ms 头吃 audio 不直接依赖 context, 整体估退化 0.02-0.03. 这是策略 1 (推理归一化) 的实证基础, 不需要重训."

### 当评委问"对复赛有什么准备"
"我们在初赛剩余的 13 天里:
1. T1 推理归一化 (已实现 + 实测, slide N)
2. T2 train 模拟变长重训 4 base (slide N+1, 估再降 50%)
3. T3 cross-context probe 表 (slide N+2, 答辩这里)
4. T4 复赛 docker prototype (slide N+3)"

### 当评委追问 "为什么 10s 只跌 0.029, 5s 才 0.044"
"两个原因: 一是 LGBM 的滚动窗特征 (10/25/50/100 chunk) 在 5s 上下文里仍能提供 0/25/50/100 chunk 的部分信号; 二是我们的训练数据 (cap5 切片) 本身已包含 cap0-cap4 多窗口起点, 模型间接见过非起始位置的窗口. 这是 D-25 双 SSL_ms 协同设计**意外地**对复赛动态时长更鲁棒."

## 局限性 (诚实告知)

1. N=617 cap5 窗口 BC 类只 7 个正例, BC 退化未能评估 (BC F1 全 0)
2. 这只是 ctx 单源, R4 全栈推算未实测 (需要 T1 推理归一化 + 重做 R4 推理)
3. 训练集 ≠ 测试集 2 业务领域 — 内部私有数据差异可能让退化加剧
4. "估" R4 退化 = ctx 退化 × 0.5 是粗略加权, 真实可能 ×0.3 - ×0.7
